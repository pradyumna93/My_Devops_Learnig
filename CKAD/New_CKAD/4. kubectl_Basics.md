# kubectl: The Primary Kubernetes CLI Tool

## Introduction

`kubectl` is the primary command-line interface (CLI) tool you'll use throughout your Kubernetes exam preparation and when working with Kubernetes clusters. Whether you're preparing for exams, working in local environments, or managing production systems, `kubectl` is essential for interacting with Kubernetes clusters.

## What is kubectl?

`kubectl` (pronounced "cube-control," "kube-cuddle," or simply "kubectl") serves as the **gateway to the API server**. 

### Key Points:
- **Primary Interface** - Used to interact with all Kubernetes clusters
- **API Server Wrapper** - Acts as a wrapper around HTTP-based REST API calls
- **Authentication Handling** - Internally uses your local kubeconfig file for authentication
- **Simpler Alternative** - Simplifies complex HTTP requests that would otherwise require curl or REST API calls

## Why kubectl Matters for Kubernetes Exams

Understanding `kubectl` is **mandatory** for Kubernetes exam success:
- You must know common `kubectl` commands and subcommands
- It's the primary tool for all cluster interactions
- Proficiency with `kubectl` syntax is essential for exam tasks
- Most exam operations require `kubectl` commands

## kubectl Command Structure

Every `kubectl` command follows a consistent syntax pattern:

```
kubectl <COMMAND> <TYPE> <NAME> <FLAGS>
```

### Breaking Down the Syntax:

| Component | Description | Example |
|-----------|-------------|---------|
| **kubectl** | Main command | `kubectl` |
| **COMMAND** | The action/subcommand | `get`, `create`, `delete`, `describe` |
| **TYPE** | The resource type | `pod`, `service`, `deployment`, `configmap` |
| **NAME** | Specific resource instance (optional) | `nginx-pod`, `my-service` |
| **FLAGS** | Additional options | `-n kube-system`, `-o yaml` |

### Understanding Each Component:

**1. Command (Subcommand)**
- Defines the action you want to perform
- Common commands:
  - `get` - Retrieve information about resources
  - `create` - Create new resources
  - `delete` - Delete resources
  - `describe` - Show detailed information
  - `logs` - View container logs
  - `exec` - Execute commands in containers

**2. Type (Resource Type)**
- Specifies the Kubernetes resource to interact with
- Common resource types:
  - `pod` / `pods`
  - `service` / `services`
  - `deployment` / `deployments`
  - `configmap` / `configmaps`
  - `secret` / `secrets`
  - `nodes`, `namespaces`, etc.

**3. Name (Resource Instance)**
- Specifies a particular resource
- Optional - omit to get all resources of that type
- Example: `my-nginx-pod`, `web-service`

**4. Flags (Optional Parameters)**
- Provide additional options for the command
- Common flags:
  - `-n <namespace>` - Specify namespace (default is current namespace)
  - `-o yaml` - Output in YAML format
  - `-o json` - Output in JSON format
  - `-o wide` - Extended output
  - `--all-namespaces` - Show across all namespaces

## kubectl Command Examples

### Example 1: List All Pods

```bash
kubectl get pods
```

**Breakdown:**
- Command: `get` (retrieve information)
- Type: `pods` (resource type)
- Name: (not specified - returns all pods)
- Namespace: (not specified - uses current namespace)
- **Output:** Lists all pods in the current namespace

### Example 2: Describe a Specific Pod

```bash
kubectl describe pod nginx-export
```

**Breakdown:**
- Command: `describe` (show detailed information)
- Type: `pod` (resource type)
- Name: `nginx-export` (specific pod)
- **Output:** Detailed information about the specific pod

### Example 3: Get Services with Namespace and YAML Output

```bash
kubectl get service -n kube-system -o yaml
```

**Breakdown:**
- Command: `get` (retrieve information)
- Type: `service` (resource type)
- Flags: `-n kube-system` (namespace flag), `-o yaml` (output format)
- **Output:** All services in the kube-system namespace in YAML format

## kubectl vs API Calls vs curl

### 1. kubectl Command

```bash
kubectl logs nginx-pod
```

Simple, straightforward command.

### 2. Equivalent API Call

```
GET /api/v1/namespaces/<namespace>/pods/<pod-name>/logs
```

More complex path structure required.

### 3. Equivalent curl Command

```bash
curl --cacert ca.crt --cert client.crt --key client.key \
  --header "Authorization: Bearer $token" \
  https://api-server:6443/api/v1/namespaces/default/pods/nginx-pod/logs
```

**Much more complex** - requires:
- Certificate management
- Token authentication
- Proper headers
- Full URL construction

### Why kubectl is Better:

- **Simpler Syntax** - Easy to remember and type
- **Built-in Authentication** - Uses kubeconfig automatically
- **Less Error-Prone** - No manual certificate/token handling
- **Consistent Interface** - All commands follow same pattern

## Key Takeaways

1. **kubectl** is your gateway to Kubernetes clusters
2. All kubectl commands follow: `kubectl COMMAND TYPE NAME FLAGS`
3. Omit the NAME to get all resources of that type
4. Use FLAGS for namespace, output format, and other options
5. kubectl handles authentication using your kubeconfig file internally
6. It's essentially a user-friendly wrapper around REST API calls

- Familiarize yourself with resource types you'll encounter

## Quick Reference

| Task | Command |
|------|---------|
| List all pods | `kubectl get pods` |
| Get pods in namespace | `kubectl get pods -n <namespace>` |
| Describe a pod | `kubectl describe pod <pod-name>` |
| Get YAML of a pod | `kubectl get pod <pod-name> -o yaml` |
| Get all resources | `kubectl get all` |
| Get services | `kubectl get services` |
| View pod logs | `kubectl logs <pod-name>` |
| Execute command in pod | `kubectl exec -it <pod-name> -- <command>` |
