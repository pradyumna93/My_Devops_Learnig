# Worker Nodes in Kubernetes

## Introduction

Worker Nodes are the backbone of Kubernetes clusters where actual application workloads run. While the Control Plane makes decisions and manages the cluster, Worker Nodes execute those decisions by running your containerized applications. Understanding Worker Node components is essential for CKAD exam success.

### Worker Node Purpose

```
Control Plane: "Deploy 3 replicas of nginx on worker nodes"
Worker Nodes: "Executing that request - running nginx containers"
```

---

## Worker Node Overview

### Key Characteristics

- **Purpose:** Host application Pods and execute containers
- **Quantity:** At least 1 required (but 3+ recommended for production)
- **Workload:** Runs the actual containerized applications
- **Communication:** Reports status back to Control Plane
- **Components:** 3 main components (Kubelet, kube-proxy, Container Runtime)

### Worker Node Architecture

## Worker Node Components

Worker nodes consist of three essential components that work together to run and manage containerized workloads.

### 1. Kubelet

**Role:** Node Agent - Primary component responsible for Pod management

**Primary Responsibilities:**

1. **API Server Communication**
   - Continuously watches the API Server for new Pod assignments
   - Receives Pod specifications from the API Server
   - Reports node status and Pod status back to API Server

2. **Pod Management**
   - Ensures containers are running in Pods
   - Starts Pods assigned to the node
   - Stops Pods when they're deleted
   - Restarts failed containers (if restart policy allows)

3. **Lifecycle Management**
   - Implements Pod lifecycle hooks (startup, readiness, liveness)
   - Handles Pod eviction
   - Manages storage volumes for Pods
   - Enforces resource limits

4. **Container Runtime Interaction**
   - Works closely with the Container Runtime
   - Directs container start/stop operations
   - Manages container images

5. **Node Reporting**
   - Reports node capacity and allocatable resources
   - Reports node conditions (Ready, MemoryPressure, DiskPressure, etc.)
   - Sends heartbeat signals to API Server

### Kubelet Workflow

```
1. Start kubelet service on node

2. Kubelet registers with API Server
   "I'm a node, here's my capacity and resources"

3. Kubelet watches API Server
   "Are there any Pods assigned to me?"

4. API Server tells Kubelet
   "Run this Pod specification on your node"

5. Kubelet talks to Container Runtime
   "Pull this image and start this container"

6. Container Runtime executes
   Container is pulled and started

7. Kubelet monitors container
   Checks if it's healthy, running, etc.

8. Kubelet reports to API Server
   "Pod is running, containers are healthy"

9. If container crashes
   Kubelet restarts it (based on restart policy)
   Reports crash to API Server

10. If Pod is deleted
    Kubelet stops the containers
    Cleans up resources
    Reports deletion to API Server
```

### Key Kubelet Points

- **Most Critical Component** - If Kubelet fails, no containers can run on that node
- **Bridge Between Planes** - Connects Control Plane with container execution
- **Status Reporter** - All node and Pod status comes through Kubelet
- **Always Running** - Kubelet is a daemon that continuously watches for work

---

### 2. kube-proxy

**Role:** Network Proxy - Manages network communication and Service routing

**Network Responsibilities:**

1. **Service Abstraction Implementation**
   - Makes Services work by maintaining network rules
   - Routes traffic to correct Pod backends
   - Handles Service load balancing

2. **Network Rules Maintenance**
   - Creates and maintains iptables rules on the node
   - Routes traffic based on Service definitions
   - Updates rules when Services or Pods change

3. **Inter-Node Communication**
   - Handles communication between Worker Nodes
   - Enables Pod-to-Pod communication across nodes
   - Manages network policies

4. **Load Balancing**
   - Distributes traffic among Pod replicas
   - Implements different Service types:
     - ClusterIP (internal only)
     - NodePort (external access)
     - LoadBalancer (cloud load balancer)

### How kube-proxy Works

**Service Routing Example:**

```
External Client connects to Service IP: 10.0.0.1:80

kube-proxy intercepts the connection:
- Looks up Service definition in API Server
- Finds backend Pods: 10.1.0.5, 10.1.0.6, 10.1.0.7
- Selects one Pod (using load balancing algorithm)
- Routes traffic to that Pod: 10.1.0.5:8080

Return traffic:
- Response from Pod
- kube-proxy translates back to Service IP
- Client receives response

If Pod fails:
- Kubelet detects failure
- Reports to API Server
- kube-proxy updates routing rules
- Traffic reroutes to remaining Pods
```

### kube-proxy Implementations

kube-proxy can use different backends:
- **iptables** (default) - Linux kernel packet filtering
- **IPVS** - Linux Virtual Server (better performance for many services)
- **userspace** (legacy, rarely used)

### Key kube-proxy Points

- **Network Intelligence** - Enables Services to work
- **Automatic Updates** - Rules update when Pods/Services change
- **Node-Level Component** - Runs on every node
- **Transparent to Applications** - Apps don't know about kube-proxy

---

### 3. Container Runtime

**Role:** Container Execution Engine - Runs and manages containers

**Responsibilities:**

1. **Image Management**
   - Pulls container images from registries
   - Manages local image cache
   - Handles image authentication

2. **Container Lifecycle**
   - Starts containers as directed by Kubelet
   - Stops containers
   - Manages container processes
   - Handles container restart (in cooperation with Kubelet)

3. **Resource Management**
   - Allocates resources (CPU, memory) to containers
   - Enforces resource limits
   - Manages container logging

4. **Container Isolation**
   - Uses Linux namespaces for isolation
   - Uses cgroups for resource limiting
   - Ensures containers are isolated from each other

### Supported Container Runtimes

| Runtime | Status | Notes |
|---------|--------|-------|
| **Docker** | Deprecated (as of v1.20) | Still works but no longer recommended |
| **containerd** | Recommended | Modern, lightweight, CNCF project |
| **CRI-O** | Recommended | Kubernetes-focused, CNCF project |
| **rkt** | Archived | Legacy, not recommended |

### Container Runtime Communication

```
Kubelet ─────────────────────────► Container Runtime
         "Start container with:"
         - Image: nginx:1.19
         - Ports: 80
         - Env vars: APP=web
         - Limits: 512Mi memory, 250m CPU
         
Container Runtime processes:
1. Pull image nginx:1.19
2. Create container with specifications
3. Start container process
4. Attach to container's stdout/stderr
5. Keep container running

Kubelet ◄────────────────────────── Container Runtime
        "Container status:"
        - State: running
        - PID: 12345
        - Status: healthy
```

### Container Runtime Examples

**Docker Command Equivalent:**
```bash
# What Kubelet might tell Container Runtime:
docker run --name pod-container \
  --image=nginx:1.19 \
  --memory=512Mi \
  --cpus=0.25 \
  nginx
```

### Key Container Runtime Points

- **Actual Executor** - Does the real work of running containers
- **Kubelet Dependent** - Cannot work without Kubelet directing it
- **Image Puller** - Handles all image management
- **Resource Enforcer** - Ensures containers don't exceed limits

## Worker Node Components Interaction

### How the Three Components Work Together

```
API Server (Control Plane)
    ↓
    └─────────────────────────────────────┐
                                          │
                                    Kubelet
                                          │
                                    ┌─────┴─────┐
                                    │           │
                            Container Runtime  kube-proxy
                                    │           │
                                    │           │
                    ┌───────────────┘           │
                    │ (request to start)        │
                    ↓                           │
                 Containers                    │
                 in Pod                        │
                    ↓                           │
                    └──────────────┐            │
                                   │ traffic routing
                                   ↓
                              Network Rules
```

### Complete Workflow

1. **Kubelet watches API Server** - Ready for work
2. **API Server sends Pod assignment** - "Run this Pod"
3. **Kubelet receives assignment** - Gets Pod specification
4. **Kubelet → Container Runtime** - "Pull image and start container"
5. **Container Runtime pulls image** - Downloads from registry
6. **Container Runtime starts container** - Execution begins
7. **kube-proxy creates network rules** - Traffic can flow to container
8. **Kubelet monitors container** - Checks health status
9. **Kubelet reports status** - "Container is running"
10. **API Server updates etcd** - Persists state

## Detailed Workflow: Running a Container

### Phase 1: Pod Assignment

```
Scheduler Decision:
- Pod requires: 512Mi memory, 250m CPU
- Node 1 has: 1Gi available memory, 500m CPU → ✓ Can fit
- Node 2 has: 256Mi available memory, 100m CPU → ✗ Cannot fit
- Decision: Assign to Node 1
```

### Phase 2: Container Execution

```
Kubelet on Node 1 receives assignment:
1. Retrieves Pod spec: nginx:1.19 image, port 80, 512Mi memory
2. Checks if image is locally cached - Not found
3. Instructs Container Runtime to pull image
   Container Runtime: docker pull nginx:1.19
4. Image pulled successfully
5. Instructs Container Runtime to start container
   Container Runtime: docker run -m 512M nginx:1.19
6. Container starts and runs
7. Kubelet mounts volumes (if any)
8. Kubelet configures environment variables
```

### Phase 3: Network Setup

```
kube-proxy receives Pod assignment:
1. Gets Pod IP: 10.1.0.5
2. Gets Service IP (if Pod is behind a Service): 10.0.0.1
3. Creates iptables rule:
   "Traffic to 10.0.0.1:80 → Route to 10.1.0.5:8080"
4. Updates routing tables
5. Network is now functional
```

### Phase 4: Monitoring and Reporting

```
Kubelet continuously:
1. Checks container is running
2. Monitors resource usage
3. Performs health checks (if configured)
4. Reports status periodically to API Server
5. If container crashes:
   - Restarts it (based on restart policy)
   - Reports crash to API Server
6. If Pod is deleted:
   - Stops container
   - Cleans up resources
   - Updates API Server
```

## Worker Node Management Operations

### Node Status Conditions

Nodes can have various status conditions:

| Condition | Meaning | Status |
|-----------|---------|--------|
| **Ready** | Node is healthy and ready for Pods | True/False |
| **MemoryPressure** | Node is low on memory | True/False |
| **DiskPressure** | Node is low on disk space | True/False |
| **PIDPressure** | Node is running too many processes | True/False |
| **NetworkUnavailable** | Node network is not ready | True/False |

### Checking Node Status

```bash
kubectl get nodes
# Shows all nodes and Ready status

kubectl describe node worker-1
# Shows detailed node information, conditions, capacity, allocatable resources

kubectl top nodes
# Shows resource usage on nodes
```

### Common Node Issues

| Issue | Cause | Resolution |
|-------|-------|-----------|
| **NotReady** | Kubelet not running or API communication lost | Check Kubelet service, network connectivity |
| **MemoryPressure** | Node memory exhausted | Add memory or remove Pods |
| **DiskPressure** | Node disk full | Free disk space or add storage |
| **Kubelet failure** | Kubelet service crashed | Restart Kubelet, check logs |
| **Container Runtime failure** | Docker/containerd crashed | Restart container runtime |

## Worker Node Lifecycle

### Node Addition

```
1. Physical/Virtual machine created
2. OS installed
3. Kubelet service installed and started
4. Container runtime installed and started
5. Kubelet contacts API Server
6. API Server registers node
7. Node appears in "kubectl get nodes"
8. Scheduler can assign Pods to this node
```

### Node Removal

```
1. Cordon node: kubectl cordon node-name
   - Node marked as unschedulable
   - No new Pods assigned
   - Existing Pods continue running

2. Drain node: kubectl drain node-name
   - Evicts all Pods gracefully
   - Pods are rescheduled to other nodes
   - Node becomes empty

3. Delete node: kubectl delete node node-name
   - Node removed from cluster
4. Stop/terminate machine (if needed)
```

## Understanding Pod and Container Execution

### Pod vs Container vs Container Runtime

```
Pod (Kubernetes concept)
├── Container 1
│   └── Process 1 (running in Container Runtime)
├── Container 2
│   └── Process 2 (running in Container Runtime)
└── Container N
    └── Process N (running in Container Runtime)

All containers share:
- Network namespace (same IP address)
- Storage volumes (if specified)
- IPC namespace (if specified)
```

### Example: Running nginx Pod

```yaml
Pod Specification:
- name: nginx-pod
- containers:
  - name: nginx
    image: nginx:1.19
    ports: 80
    memory: 512Mi

Execution:
1. Kubelet receives this specification
2. Kubelet → Container Runtime: "Pull nginx:1.19"
3. Container Runtime downloads image
4. Container Runtime creates container
5. Container Runtime starts nginx process
6. Kubelet monitors the nginx process
7. kube-proxy sets up port 80 routing
8. Traffic flows to nginx container
```

## Important Points for Exam Success

1. **Kubelet is critical** - If it fails, nothing runs on that node
2. **Container Runtime actually executes** - Kubelet directs it
3. **kube-proxy enables Services** - Without it, Services don't route traffic
4. **All three must work together** - Failure of any breaks functionality
5. **Kubelet reports status** - All node info comes from Kubelet
6. **Scheduler assigns to nodes** - But Kubelet actually starts Pods
7. **Network rules are node-local** - kube-proxy maintains them on each node

## Common Exam Scenarios

### Scenario 1: Pod won't start on a node
**Check:**
1. Is Kubelet running? `systemctl status kubelet`
2. Is Container Runtime running? `systemctl status docker`
3. Does node have enough resources?
4. Are there taints on the node?

### Scenario 2: Service isn't routing traffic
**Check:**
1. Is kube-proxy running on nodes?
2. Are iptables rules correct?
3. Are Pods running?
4. Is Service selector correct?

### Scenario 3: Node is NotReady
**Check:**
1. Kubelet running and healthy?
2. Network connectivity to API Server?
3. Disk space available?
4. Memory pressure?

## Key Takeaways

1. Worker Nodes host and run application containers
2. **Kubelet** is the node agent connecting to Control Plane
3. **Container Runtime** executes the actual containers
4. **kube-proxy** handles all networking and routing
5. Three components must work together seamlessly
6. Kubelet reports all node and Pod status
7. Container Runtime pulls images and starts processes
8. kube-proxy ensures Services route traffic correctly
9. Node failures are handled by controllers in Control Plane
10. Every node must have all three components to function
