# Kubernetes Architecture

## Introduction

Understanding Kubernetes architecture is fundamental to exam success and working with Kubernetes effectively. The architecture consists of two major categories of systems: the **Control Plane** and the **Worker Nodes**. Each plays a distinct role in managing and running the cluster.

## Kubernetes Architecture Components

### Two Major Categories

| Component | Purpose | Responsibility |
|-----------|---------|-----------------|
| **Control Plane** | Management | Manage cluster, make decisions, orchestrate workloads |
| **Worker Nodes** | Execution | Run application containers, execute workloads |

### Analogy

Think of it like a restaurant:
- **Control Plane** = Management/Chefs (decide what to cook, where, how many)
- **Worker Nodes** = Kitchen Staff (actually cook and prepare the food)

---

## Control Plane Components

The Control Plane is responsible for managing the entire Kubernetes cluster. All decision-making, scheduling, and cluster management happens here.

### 1. API Server (Kubernetes API)

**Role:** Central hub of the entire Kubernetes cluster

**Responsibilities:**
- Primary interface for all cluster interactions
- Processes all requests from users and components
- Direct access to etcd database
- Authenticates and authorizes requests
- Coordinates all cluster activities
- Acts as the gateway to the cluster

**Key Points:**
- Every operation in the cluster goes through the API Server
- All components communicate via the API Server
- It's the only component that directly accesses etcd
- Essential for cluster operation - if it goes down, cluster management stops

### 2. etcd (Distributed Key-Value Store)

**Role:** Persistent storage for all cluster data

**Stores:**
- Cluster state and configuration
- Resource definitions (Pods, Services, Deployments, etc.)
- Configuration data for all resources
- Metadata about nodes and components
- All persistent cluster information

**Characteristics:**
- Distributed key-value database
- Reliable and consistent
- Single instance or multiple instances (for high availability)
- Only the API Server should access etcd directly

**Deployment Options:**
- Single instance (simple, but single point of failure)
- Multiple instances (high availability, better consistency and availability)

### 3. Scheduler (kube-scheduler)

**Role:** Pod placement and assignment to nodes

**Responsibilities:**
- Watches for newly created Pods that haven't been assigned to a node
- Selects the most appropriate node for each Pod based on:
  - Resource requirements (CPU, memory)
  - Pod constraints and affinity rules
  - Resource availability on nodes
  - Taints and tolerations
  - Other scheduling policies

**How It Works:**
1. Pod is created and submitted to API Server
2. Scheduler watches for unscheduled Pods
3. Scheduler evaluates all nodes based on requirements
4. Scheduler assigns Pod to the best-fit node
5. Kubelet on that node receives the assignment and starts the Pod

**Important:** Scheduler only assigns Pods to nodes; it doesn't actually start them.

### 4. Controller Manager (kube-controller-manager)

**Role:** Maintains desired cluster state through various controllers

**Purpose:** Ensures all resources run in their desired state

**How It Works:**
- Runs multiple controller processes
- Each controller manages specific resource types
- Continuously monitors resources
- Takes corrective actions if reality differs from desired state

**Common Controllers:**

| Controller | Manages |
|------------|---------|
| **Node Controller** | Monitors node health, handles node failures |
| **Deployment Controller** | Manages Deployments and ReplicaSets |
| **Job Controller** | Manages batch Jobs |
| **Service Account Controller** | Manages ServiceAccounts and tokens |
| **Service Controller** | Manages Services and their endpoints |
| **Endpoints Controller** | Manages Endpoints |
| **StatefulSet Controller** | Manages StatefulSets |
| **DaemonSet Controller** | Manages DaemonSets |

**Self-Healing Example:**
```
Desired State: 3 Pod replicas running
Current State: 2 pods running (1 crashed)
Action: Controller creates 1 new Pod to match desired state
```

**Custom Controllers:** You can create custom controllers for domain-specific logic

### 5. Cloud Controller Manager (cloud-controller-manager)

**Role:** Interface between Kubernetes and cloud providers

**Handles Cloud-Specific Operations:**
- **Load Balancing** - Creates and manages cloud load balancers
- **Storage** - Manages cloud storage provisioning
- **Networking** - Manages cloud network configurations
- **Node Management** - Interacts with cloud provider node APIs

**When Used:**
- Only relevant when running Kubernetes on cloud platforms (AWS, Azure, GCP, etc.)
- Not needed for on-premise or local clusters
- Enables Kubernetes to leverage cloud-native features

### Control Plane Summary

```
API Server
    ↓
  (hub)
   ↙ ↓ ↘
etcd  Scheduler  Controller Manager  Cloud Controller
                                      Manager
```

## Worker Node Components

Worker Nodes are responsible for running your application workloads. Each worker node has three main components.

### 1. Kubelet

**Role:** Agent running on each worker node, communicates with Control Plane

**Responsibilities:**
- Communicates with API Server
- Receives Pod assignments from Scheduler (via API Server)
- Starts and stops containers
- Ensures containers are running and healthy
- Reports node status back to API Server
- Implements pod lifecycle hooks
- Runs kubelet to ensure containers match desired state

**How It Works:**
1. Kubelet watches the API Server for new Pod assignments
2. Kubelet receives Pod specification
3. Kubelet pulls container images
4. Kubelet instructs container runtime to start containers
5. Kubelet monitors container health
6. Kubelet reports status back to API Server

**Key Point:** Kubelet is the most important component on each worker node

### 2. Container Runtime

**Role:** Executes and manages containerized applications

**Responsibilities:**
- Pulls container images from registries
- Runs containers
- Manages container lifecycle (start, stop, restart)
- Handles container resource allocation

**Supported Container Runtimes:**
- Docker
- containerd
- CRI-O
- Other CRI-compatible runtimes

**Note:** Docker was deprecated in Kubernetes but other runtimes (containerd, CRI-O) are widely used

**Example:** When Kubelet tells the Container Runtime to start an nginx container:
```
Kubelet → Container Runtime → Start nginx container
```

### 3. kube-proxy

**Role:** Network proxy maintaining network rules on nodes

**Responsibilities:**
- Manages network rules on the node
- Routes network traffic between Services and Pods
- Implements Service abstraction (ClusterIP, NodePort, LoadBalancer)
- Maintains network connectivity within the cluster
- Handles network policies

**How It Works:**
```
External Traffic → Service → kube-proxy → Routes to Pod IP
```

**Example:**
```
User connects to Service at IP 10.0.0.1:80
kube-proxy intercepts the connection
kube-proxy routes traffic to actual Pod IP 10.1.0.5:8080
```

### Worker Node Summary

```
Worker Node
├── Kubelet (Communication)
├── Container Runtime (Execution)
└── kube-proxy (Networking)
```

## Component Interactions and Communication Flow

### Request Flow: Creating a Pod

```
User/kubectl
    ↓
API Server (validates request)
    ↓
etcd (stores Pod definition)
    ↓
Scheduler (watches for unscheduled pods)
    ↓
Scheduler assigns Pod to Node
    ↓
API Server updates Pod assignment
    ↓
Kubelet (watches API Server for assignments)
    ↓
Kubelet receives Pod assignment
    ↓
Container Runtime (pulls image, starts container)
    ↓
Pod is running
    ↓
Kubelet reports status to API Server
    ↓
API Server updates etcd
```

### Component Dependency on API Server

All components communicate through the API Server:

```
Controller Manager → API Server ← Scheduler
                       ↓
                    etcd
                       ↓
                   Kubelet (on each node)
                       ↓
                Container Runtime
```

## Key Architectural Concepts

### Control Plane vs Worker Nodes

| Aspect | Control Plane | Worker Node |
|--------|---------------|------------|
| **Purpose** | Manage cluster | Run workloads |
| **Components** | API Server, etcd, Scheduler, Controllers | Kubelet, Container Runtime, kube-proxy |
| **Decision Making** | Yes (scheduling, scaling) | No (executes decisions) |
| **Runs Pods** | Can run (but typically doesn't) | Yes (runs application Pods) |
| **Failures** | Affects cluster management | Affects workload availability |

### High Availability Considerations

**Control Plane HA:**
- Multiple API Server instances
- Multiple etcd instances with distributed consensus
- Multiple Scheduler instances
- Multiple Controller Manager instances

**Worker Node HA:**
- Multiple worker nodes
- Pod replicas across nodes
- Services for load balancing

### Minimal Cluster Setup

```
Minimum 1 Control Plane + 1 Worker Node = Functional Cluster
```

But for production:
```
3+ Control Plane Nodes (HA) + 3+ Worker Nodes
```

## Data Flow Examples

### Example 1: Deploying an Application

```
1. User runs: kubectl create deployment my-app --image=nginx

2. kubectl sends request to API Server

3. API Server:
   - Validates request
   - Stores in etcd
   - Creates Deployment object

4. Deployment Controller watches for new Deployments
   - Creates ReplicaSet
   - ReplicaSet Controller creates Pods

5. Scheduler watches for unscheduled Pods
   - Assigns each Pod to a suitable Worker Node

6. On Worker Nodes, Kubelet:
   - Sees Pod assignment
   - Pulls nginx image
   - Instructs Container Runtime to start container

7. Pods are running, application is live
```

### Example 2: Scaling an Application

```
1. User runs: kubectl scale deployment my-app --replicas=3

2. API Server updates Deployment in etcd

3. Deployment Controller:
   - Sees replicas changed from 1 to 3
   - Updates ReplicaSet to 3 replicas

4. ReplicaSet Controller:
   - Creates 2 new Pods to reach desired state

5. Scheduler:
   - Assigns new Pods to available Worker Nodes

6. Kubelets on Worker Nodes:
   - Start the new containers

7. Three replicas are running
```

### Example 3: Node Failure Recovery

```
1. Worker Node crashes (network failure)

2. Kubelet cannot report status (no heartbeat)

3. Node Controller (part of Controller Manager):
   - Detects missed heartbeat after timeout period
   - Marks node as NotReady

4. Pod Eviction Controller:
   - Evicts Pods from the failed node

5. Scheduler:
   - Schedules Pod replicas on other healthy nodes

6. New Pods start on healthy nodes

7. Application remains available (self-healing)
```

## Important Points to Remember

### For Exam Preparation

1. **API Server is central** - All communication flows through it
2. **Scheduler doesn't start pods** - It only assigns them to nodes
3. **Kubelet actually starts containers** - Kubelet communicates with container runtime
4. **etcd stores everything** - All cluster state and configuration
5. **Controllers maintain state** - They continuously work to match desired state
6. **kube-proxy handles networking** - Services work because of kube-proxy

### Common Misconceptions

- ❌ Scheduler starts Pods → ✅ Scheduler only assigns Pods to nodes
- ❌ API Server runs containers → ✅ Container runtime (via Kubelet) runs containers
- ❌ Kubelet directly accesses etcd → ✅ Only API Server accesses etcd directly
- ❌ Controller Manager runs Pods → ✅ Controller Manager maintains state

## Architecture Best Practices

1. **Production Setup:**
   - 3+ Control Plane nodes (odd number for consensus)
   - 3+ Worker nodes minimum
   - Separate etcd cluster for high availability

2. **Resource Allocation:**
   - Control Plane needs adequate CPU/memory
   - Worker nodes sized based on workload requirements

3. **Monitoring:**
   - Monitor all Control Plane components
   - Monitor Node health
   - Monitor etcd database health and performance

4. **Security:**
   - Secure API Server access
   - etcd should be encrypted
   - Use RBAC for authorization

## Key Takeaways

1. Kubernetes has two main layers: **Control Plane** (management) and **Worker Nodes** (execution)
2. **API Server** is the central hub connecting all components
3. **etcd** stores all cluster data persistently
4. **Scheduler** assigns Pods to nodes (doesn't start them)
5. **Kubelet** actually runs containers on each node
6. **Controller Manager** maintains desired state through various controllers
7. **kube-proxy** handles networking and Service routing
8. **Cloud Controller Manager** provides cloud provider integration
9. All components communicate through the **API Server**
10. Understanding this architecture is fundamental to Kubernetes
