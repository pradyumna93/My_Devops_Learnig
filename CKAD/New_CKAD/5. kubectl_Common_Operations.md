# Common kubectl Operations and Commands

## Introduction

Beyond basic kubectl syntax, there are several important operational commands that will help you understand your cluster, get help when needed, and learn about available resources. These commands are essential for exam preparation and daily cluster management.

---

## 1. kubectl version

### Purpose
Display version information about both the kubectl client and the Kubernetes API server.

### Syntax
```bash
kubectl version
```

### What It Shows:
- **Client Version** - Version of the kubectl CLI tool installed on your machine
- **Server Version** - Version of the API server running in the Kubernetes cluster

### Example Output:
```
Client Version: v1.28.0
Server Version: v1.27.5
```

### When to Use:
- Verify that client and server versions are compatible
- Check cluster version before installing new applications
- Troubleshoot version-related issues

---

## 2. kubectl cluster-info

### Purpose
Display information about the Kubernetes cluster and its control plane.

### Syntax
```bash
kubectl cluster-info
```

### What It Shows:
- **Kubernetes Master/Control Plane** - The main control plane endpoint
- **Hostname** - The address where the cluster is running
- **Port** - The port on which the cluster is accessible (typically 6443)
- **CoreDNS** - The DNS service information
- **Other addons** - Information about cluster addons

### Example Output:
```
Kubernetes control plane is running at https://api-server-hostname:6443
CoreDNS is running at https://api-server-hostname:6443/api/v1/...
```

### When to Use:
- Understand cluster topology
- Get connection details for the API server
- Verify cluster is running
- Troubleshoot connectivity issues

## 3. kubectl api-resources

### Purpose
Display all available resource types and their properties in the Kubernetes cluster.

### Syntax
```bash
kubectl api-resources
```

### What It Shows:

| Column | Description |
|--------|-------------|
| **NAME** | Full resource name (e.g., `pods`, `services`) |
| **SHORTNAMES** | Abbreviated notation for the resource |
| **APIVERSION** | The API version used by the resource |
| **NAMESPACED** | Whether the resource is namespace-scoped (true) or cluster-scoped (false) |
| **KIND** | The resource kind |

### Understanding Shortnames

Shortnames allow you to use abbreviated commands instead of typing full resource names:

| Full Name | Shortname | Example Usage |
|-----------|-----------|----------------|
| `pods` | `po` | `kubectl get po` |
| `services` | `svc` | `kubectl get svc` |
| `deployments` | `deploy` | `kubectl get deploy` |
| `configmaps` | `cm` | `kubectl get cm` |
| `persistentvolumeclaims` | `pvc` | `kubectl get pvc` |
| `persistentvolumes` | `pv` | `kubectl get pv` |
| `replicasets` | `rs` | `kubectl get rs` |
| `daemonsets` | `ds` | `kubectl get ds` |
| `statefulsets` | `sts` | `kubectl get sts` |

### Why Shortnames Matter:

Instead of typing the lengthy command:
```bash
kubectl get persistentvolumeclaims
```

You can simply use:
```bash
kubectl get pvc
```

This is especially useful in exams where typing speed and accuracy matter, and with complex resource names like `persistentvolumeclaims`.

### When to Use:
- Discover available resources in your cluster
- Learn shortnames for commonly used resources
- Check if a resource is namespace-scoped or cluster-scoped
- Understand the API version for each resource type

## 4. kubectl help

### Purpose
Access built-in help documentation for kubectl commands and subcommands.

### Syntax Variations:

#### General Help
```bash
kubectl -h
# or
kubectl --help
```

Shows all available kubectl commands organized by category (basic, deployment, cluster management, troubleshooting, etc.)

#### Help for Specific Subcommand
```bash
kubectl <COMMAND> -h
# or
kubectl <COMMAND> --help
```

### Examples:

**Get help for the `get` command:**
```bash
kubectl get -h
```
Shows:
- All flags available with the `get` command
- Common usage examples
- Output format options

**Get help for the `create` command:**
```bash
kubectl create -h
```
Shows:
- Subcommands available (e.g., `cluster-role`, `config-map`, `secret`)
- Usage patterns
- Available flags

**Get help for nested subcommands:**
```bash
kubectl create configmap -h
```
Shows:
- How to create a ConfigMap
- Required and optional flags
- Examples

### Organized Help Categories:

When you run `kubectl -h`, help is organized by:
- **Basic Commands** - `create`, `expose`, `run`, `set`
- **Deployment Commands** - `rollout`, `scale`, `autoscale`
- **Cluster Management** - `certificate`, `cluster-info`, `top`
- **Troubleshooting** - `describe`, `logs`, `exec`, `port-forward`

### When to Use:
- You're unfamiliar with a command's syntax
- You need to see available flags
- You want examples of command usage
- You're learning new kubectl operations
- During exams when you need quick syntax reminders

## 5. kubectl explain

### Purpose
Get detailed documentation about Kubernetes resource properties and structure, directly from the cluster.

### Syntax
```bash
kubectl explain <RESOURCE>
kubectl explain <RESOURCE.FIELD>
kubectl explain <RESOURCE.FIELD.NESTED_FIELD>
```

### Basic Usage

**Example: Understand Pod structure**
```bash
kubectl explain pod
```

**Output Includes:**
- **KIND** - The resource type (e.g., Pod)
- **VERSION** - API version (e.g., v1)
- **DESCRIPTION** - What the resource is
- **FIELDS** - All available properties with their types and descriptions

### Nested Properties

You can drill down into nested properties using dot notation:

**Example 1: Understand Pod spec**
```bash
kubectl explain pod.spec
```
Shows all properties available under `spec` in a pod definition.

**Example 2: Understand Pod spec containers**
```bash
kubectl explain pod.spec.containers
```
Shows the structure of containers within the spec.

**Example 3: Understand even deeper nesting**
```bash
kubectl explain pod.spec.containers.resources
```
Shows resource limits and requests structure.

### Real-World Example Scenario:

You're writing a pod YAML manifest but unsure about the exact field structure:

```bash
# Understand the Pod structure
kubectl explain pod

# Understand what goes in spec
kubectl explain pod.spec

# Understand container properties
kubectl explain pod.spec.containers

# Understand resource limits
kubectl explain pod.spec.containers.resources

# Understand environment variables
kubectl explain pod.spec.containers.env
```

### When to Use:
- Writing YAML manifests and unsure of field names
- Learning about new resource types
- Figuring out required vs optional fields
- Understanding nested property structures
- Troubleshooting manifest validation errors
- **During exams** - when you need to verify manifest syntax

### Exam Tip:
The `kubectl explain` command is invaluable during Kubernetes exams because:
- It helps you write correct YAML syntax
- It shows mandatory vs optional fields
- It provides sample values
- It eliminates guesswork about complex nested structures

## Practical Exam Scenarios

### Scenario 1: Creating a Pod with Environment Variables
```bash
# You forgot the exact syntax for environment variables
kubectl explain pod.spec.containers.env

# This will show you the structure:
# - name: string (required)
# - value: string
# - valueFrom: EnvVarSource
```

### Scenario 2: Setting Resource Limits
```bash
# You need to know the correct field structure
kubectl explain pod.spec.containers.resources

# This shows you:
# limits: ResourceList
# requests: ResourceList
```

### Scenario 3: Checking Available Shortnames
```bash
# You want to use shorthand for deployments
kubectl api-resources | grep deployment

# Shows: deployments, deploy
# So you can use: kubectl get deploy
```

## Key Takeaways

1. **kubectl version** - Verify client and server compatibility
2. **kubectl cluster-info** - Get cluster connection details
3. **kubectl api-resources** - Discover resources and their shortnames
4. **kubectl help (-h)** - Get syntax and usage examples
5. **kubectl explain** - Understand resource structure and fields
6. **Shortnames** - Speed up typing with abbreviated resource names
7. **Dot notation** - Use dots to navigate nested properties with explain

- `kubectl explain` is your best friend when writing YAML during the exam
