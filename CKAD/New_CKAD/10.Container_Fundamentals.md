# 🐳 Container Fundamentals

This note summarizes the high-level concepts of containers, why they are important, and how they relate to Kubernetes and container runtimes.

---

## 1. What is a Container?

A container packages an application together with:

- its runtime environment
- its dependencies
- its configuration

into a single portable unit.

This makes the application behave consistently across different machines, such as:

- local development machines
- testing environments
- production servers

---

## 2. Why Containers Exist

In traditional deployments, an application may work on one machine but fail on another because of:

- missing dependencies
- different software versions
- environment-specific configuration

Containers solve this problem by bundling everything the application needs into one package.

So, instead of depending on the host machine setup, the application carries its own required environment with it.

---

## 3. Why Containers Are Useful

### ✅ Consistent Environment
Containers provide a consistent runtime environment regardless of the underlying host system.

### ✅ Isolation
Each container runs in its own isolated environment, so it does not depend heavily on the host system.

### ✅ Lightweight
Containers are generally lighter than virtual machines because they share the host OS instead of carrying a full operating system inside each instance.

### ✅ Faster Deployment
Containers make it easier and faster to deploy applications to production environments.

---

## 4. Containers and Kubernetes

As applications grow, you often need many containers to run together.

Managing all these containers manually becomes difficult, so we use an orchestration tool.

Kubernetes is the most popular container orchestration platform. It helps manage:

- container lifecycle
- deployment
- scaling
- networking
- availability

---

## 5. Container Runtime

A container runtime is the software responsible for actually running containers on a machine.

When Kubernetes manages containers, it relies on container runtimes to:

- start containers
- stop containers
- monitor containers
- manage their lifecycle

So the relationship is:

- Kubernetes orchestrates the containers
- container runtimes run them
- actual applications run inside the containers

---

## 6. Popular Container Runtimes

Some common container runtimes are:

- Docker Engine
- containerd
- CRI-O

### Docker Engine
Docker is widely used and provides a complete platform for building, running, and managing containers.

### containerd
containerd is a lightweight runtime commonly used in Kubernetes environments.

### CRI-O
CRI-O is a Kubernetes-native runtime designed for Kubernetes integration.

---

## 7. Popular Container Tools

### Docker
Docker is one of the most popular tools for working with containers. It provides support for:

- building images
- running containers
- managing containers

### Podman
Podman is another container tool that provides similar functionality to Docker.

One key difference is that Podman is often used with a rootless model by default.

---

## 8. Key Takeaways

- A container packages an application with its dependencies and configuration.
- Containers provide consistency, isolation, and faster deployment.
- Kubernetes is used to orchestrate many containers.
- Container runtimes are required to run containers on the host system.
- Docker, containerd, and CRI-O are important runtime-related concepts in Kubernetes.

## 10. Dockerfile Basics

A Dockerfile contains the instructions used to build a container image.

It usually includes:

- a base image
- instructions for setup
- the command to run the application

### Example Dockerfile

```dockerfile
FROM python:3.11-slim
WORKDIR /app
COPY app.py /app/app.py
CMD ["python", "app.py"]
```

### What each instruction does

- `FROM` → defines the base image
- `WORKDIR` → sets the working directory inside the container
- `COPY` → copies files from the host to the container
- `CMD` → defines the default command to run when the container starts

---

## 11. Image Build Process

When Docker builds an image, it follows the instructions in the Dockerfile step by step.

The process is based on:

- the base image
- the instructions defined in the Dockerfile
- caching for faster rebuilds

If a previous build already used the same layers, Docker can reuse them and save time.

---

## 12. Basic Docker Commands

Some common commands you should know are:

```bash
docker build -t myapp .
docker run myapp
docker ps
docker images
docker logs <container_id>
docker inspect <container_id>
```

These commands are enough for most basic exam scenarios.

---

## 13. Best Practices for Dockerfiles

When writing a Dockerfile, keep these points in mind:

- use a specific and lightweight base image
- keep the number of layers small
- copy only the files needed for the application
- avoid unnecessary files to keep the image size smaller
