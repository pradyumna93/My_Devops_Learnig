# ☸️ Kubernetes Object Management Approaches

This note summarizes the three main ways to create and manage Kubernetes objects: imperative, declarative, and hybrid.

---

## 1. What Are Kubernetes Objects?

Kubernetes objects are the resources that run inside a cluster, such as:

- Pods
- Deployments
- Services
- ConfigMaps
- Secrets
- Roles
- Namespaces

These objects are managed using different approaches depending on the task.

---

## 2. The Three Approaches

### 1) Imperative Approach

The imperative approach uses the `kubectl` CLI directly to create and manage resources.

It is useful for:

- one-off tasks
- quick troubleshooting
- simple operations

Example:

```bash
kubectl run nginx --image=nginx
```

This creates a Pod named `nginx` using the `nginx` image.

### 2) Declarative Approach

The declarative approach uses YAML manifest files to define the desired state of resources.

This is the recommended approach for most real-world Kubernetes work because it is:

- repeatable
- version controllable
- easier to manage at scale

Example:

```bash
kubectl apply -f pod.yaml
```

### 3) Hybrid Approach

The hybrid approach combines both imperative and declarative methods.

For example:

- you may use `kubectl create` for a quick task
- then use YAML files for long-term management

This approach is useful when you need speed but also want maintainability.

---

## 3. Imperative Approach in Detail

The imperative approach is best when you want to quickly:

- create a resource
- update a resource
- scale a resource
- expose a service

### Common Examples

Create a Pod:

```bash
kubectl run nginx --image=nginx
```

Create a Deployment:

```bash
kubectl create deployment webapp --image=nginx
```

Scale a Deployment:

```bash
kubectl scale deployment webapp --replicas=3
```

Expose a Deployment as a Service:

```bash
kubectl expose deployment webapp --port=80
```

---

## 4. Advantages of the Imperative Approach

- fast and simple
- good for quick testing
- useful for troubleshooting
- easy to learn for beginners

---

## 5. Limitations of the Imperative Approach

- not very repeatable
- hard to version control
- less flexible for complex configurations
- commands are not stored as a desired-state definition

This means that if you want to recreate the same setup later, it is harder to do so reliably.

---

## 6. Declarative Approach in Detail

The declarative approach uses configuration files, usually YAML, to define the desired state of a Kubernetes resource.

Once the manifest is applied, Kubernetes works to ensure that the actual state matches the desired state defined in the file.

### Example

A simple Pod manifest:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
spec:
  containers:
  - name: nginx
    image: nginx:latest
```

Apply it using:

```bash
kubectl apply -f pod.yaml
```

You can verify it with:

```bash
kubectl get pod nginx-pod
```

---

## 7. Important Declarative Concept: Last Applied Configuration

When you create or update a resource using `kubectl apply`, Kubernetes stores the last applied configuration as an annotation:

```text
kubectl.kubernetes.io/last-applied-configuration
```

This is used to track the desired state and help Kubernetes apply changes correctly.

---

## 8. Desired State vs Current State

Every Kubernetes object has two key concepts:

- `spec` → the desired state you define
- `status` → the current state observed by Kubernetes

For example:

```bash
kubectl get pod nginx-pod -o yaml
```

You will see:

- `spec` with what you want the pod to be like
- `status` showing what is actually happening in the cluster

---

## 9. Three-Way Merge

When you update a resource using `kubectl apply`, Kubernetes uses a process called the three-way merge.

It compares:

1. the current state of the object
2. the new desired state from the manifest
3. the last applied configuration

This allows Kubernetes to update only the required changes instead of replacing everything blindly.

---

## 10. Advantages of the Declarative Approach

- repeatable and consistent
- easier to version control
- better for complex configurations
- ideal for infrastructure as code
- supports predictable updates and rollback

---

## 11. Limitations of the Declarative Approach

- slightly slower for simple one-off tasks
- requires understanding of YAML manifests
- less convenient for quick debugging than imperative commands

---

## 12. Why Declarative Is Preferred

The declarative approach is preferred in production because it stores the desired state in YAML files.

This provides:

- consistency
- easy updates
- easier rollback
- better collaboration through version control

---

## 13. Hybrid Approach

The hybrid approach combines both imperative and declarative techniques.

In practice, this means:

1. use an imperative command to generate or create a manifest
2. edit the YAML as needed
3. apply it declaratively to create or update the resource

### Example Workflow

Generate a Pod manifest imperatively:

```bash
kubectl run nginx --image=nginx --dry-run=client -o yaml > pod.yaml
```

Then edit the file and apply it:

```bash
kubectl apply -f pod.yaml
```

This gives you the speed of imperative commands while keeping the resource definition in a reusable YAML file.

---

## 14. When to Use Which Approach

### Use Imperative when:

- you need a quick one-off action
- you are debugging
- you want to test something fast

Example:

```bash
kubectl run nginx --image=nginx
```

### Use Declarative when:

- the YAML manifest already exists
- you want repeatable and version-controlled resources
- you are working in production or real-world environments

Example:

```bash
kubectl apply -f deployment.yaml
```

### Use Hybrid when:

- you want to generate a manifest quickly and then customize it
- you need both speed and flexibility
- the exam asks for a custom configuration

---

## 15. Exam Tip

For Kubernetes exams, remember:

- use the imperative approach for quick, one-off tasks
- use the declarative approach for managing resources in a controlled and repeatable way
- use the hybrid approach when both speed and maintainability matter

---

## 8. Quick Revision Summary

> Kubernetes resources can be managed using three approaches: imperative for quick CLI-based tasks, declarative for YAML-based desired-state management, and hybrid for combining both.
