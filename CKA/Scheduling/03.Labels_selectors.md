# Labels, Selectors, and Annotations in Kubernetes
---

## 1. What Are Labels in Kubernetes?

**Labels** are **key-value pairs** attached to Kubernetes objects such as:

* Pods
* ReplicaSets
* Deployments
* Services
* Nodes

They are used to **organize, group, and select objects**.

### Example: Pod with Labels

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:
    app: web
    tier: frontend
    env: prod
spec:
  containers:
    - name: nginx
      image: nginx
```

---

## 2. Selecting Objects Using Labels

Use **label selectors** to filter Kubernetes objects.

### Select Pods with a Specific Label

```bash
kubectl get pods --selector app=web
```

### Select Using Multiple Labels (AND condition)

```bash
kubectl get pods -l app=web,tier=frontend
```

---

## 3. Labels and Selectors in ReplicaSets (Very Important for CKA)

ReplicaSets use labels and selectors to **identify which Pods they manage**.

⚠️ **Common CKA Exam Mistake**
Confusing:

* ReplicaSet labels
* Pod template labels

---

### ReplicaSet YAML (Correct Configuration)

```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: web-rs
  labels:
    app: web
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
      tier: frontend
  template:
    metadata:
      labels:
        app: web
        tier: frontend
    spec:
      containers:
        - name: nginx
          image: nginx
```

### Key Rules

* `spec.selector.matchLabels` **must match**
* `spec.template.metadata.labels`
* If they don’t match → ReplicaSet **will not create Pods**

---

## 4. Labels and Selectors in Deployments

Deployments work the same way as ReplicaSets.

### Deployment YAML Example

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-deploy
spec:
  replicas: 2
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: nginx
          image: nginx
```

---

## 5. Labels and Selectors in Services (Extremely Important)

Services use **selectors** to determine **which Pods receive traffic**.

### Service YAML Example

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web-service
spec:
  selector:
    app: web
    tier: frontend
  ports:
    - protocol: TCP
      port: 80
      targetPort: 80
```

### How This Works

* Service selector matches Pod labels
* Traffic is routed only to matching Pods

## 6. Inspect Labels on Kubernetes Objects

### View Labels on Pods

```bash
kubectl get pods --show-labels
```

### View Labels of a Specific Pod

```bash
kubectl get pod nginx-pod -o yaml
```

---

## 7. Add or Modify Labels (CKA Practical Skill)

### Add a Label to a Pod

```bash
kubectl label pod nginx-pod version=v1
```

### Overwrite an Existing Label

```bash
kubectl label pod nginx-pod version=v2 --overwrite
```

---

## 8. Node Labels and Scheduling (Exam Relevant)

Nodes also have labels, commonly used for scheduling.

### View Node Labels

```bash
kubectl get nodes --show-labels
```

### Schedule Pod Using Node Selector

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: node-selector-pod
spec:
  nodeSelector:
    disktype: ssd
  containers:
    - name: nginx
      image: nginx
```

---

## 9. What Are Annotations in Kubernetes?

**Annotations** store **non-identifying metadata**.

They are:

* NOT used for selection
* NOT used by ReplicaSets or Services

### Annotation Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: annotated-pod
  annotations:
    build-version: "1.0.4"
    owner: "devops-team"
spec:
  containers:
    - name: nginx
      image: nginx
```

---

## 10. Labels vs Annotations (CKA Comparison)

| Feature             | Labels | Annotations |
| ------------------- | ------ | ----------- |
| Used for selection  | ✅ Yes  | ❌ No        |
| Used by Services    | ✅ Yes  | ❌ No        |
| Used by ReplicaSets | ✅ Yes  | ❌ No        |
| Used for metadata   | ❌ No   | ✅ Yes       |
| Queryable           | ✅ Yes  | ❌ No        |
