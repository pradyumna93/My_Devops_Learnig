
# Kubernetes Taints & Tolerations 

## 1. Why Do We Need Taints & Tolerations?

In a Kubernetes cluster, the **scheduler** tries to place pods on nodes automatically.
Sometimes, we want **certain nodes to accept only specific pods**.

Examples:

* Dedicated nodes for databases
* Nodes with special hardware (GPU)
* Isolating critical workloads

👉 **Taints & tolerations help control which pods can run on which nodes**

---

## 2. Basic Idea

* **Taint** → Applied on a **node**
* **Toleration** → Applied on a **pod**

🔑 A node with a taint **repels pods**
🔑 A pod with a matching toleration **can run on that node**

---

## 3. Important Rule (Very Important for CKA)

> **Taints do NOT tell a pod where to go**
> **They only tell a node which pods it can accept**

---

## 4. Default Scheduling (No Taints)

Cluster:

* Nodes: `node1`, `node2`, `node3`
* Pods: `pod-a`, `pod-b`, `pod-c`

✔️ With no taints:

* Scheduler distributes pods across all nodes

---

## 5. Tainting a Node

### Example: Prevent most pods from using `node1`

```bash
kubectl taint nodes node1 role=backend:NoSchedule
```

Now:

* `node1` rejects all pods **without toleration**
* Other nodes work normally

---

## 6. Taint Structure

```
key=value:effect
```

Example:

```
role=backend:NoSchedule
```

---

## 7. Taint Effects (Must Remember)

| Effect               | Meaning                                |
| -------------------- | -------------------------------------- |
| **NoSchedule**       | Pod will not be scheduled              |
| **PreferNoSchedule** | Scheduler avoids node (not strict)     |
| **NoExecute**        | Evicts existing pods + blocks new ones |

---

## 8. Adding Toleration to a Pod

### Pod YAML with Toleration

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: backend-pod
spec:
  containers:
  - name: app
    image: nginx
  tolerations:
  - key: "role"
    operator: "Equal"
    value: "backend"
    effect: "NoSchedule"
```

✔️ This pod **can now run on node1**

---

## 9. What Happens During Scheduling?

| Pod Type                     | Can Run on node1? |
| ---------------------------- | ----------------- |
| Pod without toleration       | ❌ No              |
| Pod with matching toleration | ✅ Yes             |

⚠️ Pod **may still run on other nodes** unless restricted further

---

## 10. NoExecute Taint Explained

### Apply NoExecute

```bash
kubectl taint nodes node1 critical=true:NoExecute
```

Effects:

* Existing pods **without toleration → evicted**
* New pods **without toleration → rejected**

---

## 11. Removing a Taint

```bash
kubectl taint nodes node1 role=backend:NoSchedule-
```

---

## 12. Master Node Taint (Exam Favorite ⭐)

Kubernetes automatically taints control-plane nodes:

```
node-role.kubernetes.io/control-plane:NoSchedule
```

Why?

* Prevents application workloads on control plane

Check it using:

```bash
kubectl describe node <control-plane-node>
```

---

## 13. Common Mistake (Very Important)

❌ Using tolerations to force pod placement
✅ Use **Node Affinity** when you want pods to choose nodes

---

## 14. When to Use What?

| Requirement                  | Use               |
| ---------------------------- | ----------------- |
| Block pods from nodes        | Taints            |
| Allow specific pods          | Tolerations       |
| Force pods to specific nodes | Node Affinity     |
| Both control + restriction   | Taints + Affinity |

---

### Exercise 4 (Concept)

❓ Why doesn’t toleration guarantee scheduling?

✔️ Because toleration only allows a node to accept a pod; it does not select the node.

---

## 16. CKA Quick Revision (Memorize)

* Taints → Nodes
* Tolerations → Pods
* NoSchedule ≠ NoExecute
* Master nodes are tainted by default
* Use Node Affinity for strict placement
