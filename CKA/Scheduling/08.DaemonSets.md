

# Kubernetes DaemonSets & CronJobs`₹`

---

## Part 1: DaemonSets

## 1. What Is a DaemonSet?

A **DaemonSet** ensures that **one copy of a pod runs on every node** in the cluster.

Key behavior:

* When a **new node is added**, a pod is automatically created on it
* When a **node is removed**, the pod on that node is deleted
* Always maintains **exactly one pod per node**

📌 Similar to ReplicaSet, but:

* ReplicaSet → fixed number of pods
* DaemonSet → one pod per node

---

## 2. Why Do We Need DaemonSets?

DaemonSets are used when a service must run on **every node**, not just some.

### Common Use Cases

1️⃣ **Monitoring agents**

* Node-level metrics collection (CPU, memory, disk)

2️⃣ **Log collectors**

* Collect logs from each node

3️⃣ **Networking components**

* Network plugins (e.g., Calico, Flannel)

4️⃣ **System services**

* `kube-proxy` runs as a DaemonSet

---

## 3. How DaemonSets Work

* You create **one DaemonSet**
* Kubernetes ensures:

  * One pod per node
  * Automatic scaling as nodes change

You do **NOT** need to:

* Manually create pods
* Update pod count

---

## 4. DaemonSet Definition Structure

DaemonSet YAML looks very similar to a ReplicaSet.

### Key Fields

| Field      | Purpose        |
| ---------- | -------------- |
| apiVersion | apps/v1        |
| kind       | DaemonSet      |
| selector   | Matches pods   |
| template   | Pod definition |

---

## 5. Example: DaemonSet YAML

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: monitoring-daemon
spec:
  selector:
    matchLabels:
      app: monitoring
  template:
    metadata:
      labels:
        app: monitoring
    spec:
      containers:
      - name: monitoring-agent
        image: monitoring-agent
```

📌 Labels in `selector` **must match** labels in `template`

---

## 6. DaemonSet Commands

### Create DaemonSet

```bash
kubectl apply -f daemonset.yaml
```

### View DaemonSets

```bash
kubectl get daemonset
```

### Describe DaemonSet

```bash
kubectl describe daemonset monitoring-daemon
```

---

## 7. How Pods Are Scheduled (Important for CKA)

### Before Kubernetes v1.12

* DaemonSets bypassed the scheduler
* Pods were placed using `nodeName`

### Kubernetes v1.12 and Later

* DaemonSets use the **default scheduler**
* Use **node affinity and taints/tolerations**
* Fully integrated with scheduling logic

📌 This allows:

* Better placement control
* Resource-aware scheduling

---

## 8. Controlling Where DaemonSet Runs

DaemonSets respect:

* Node labels
* Node affinity
* Taints & tolerations

Example:

* Run DaemonSet only on worker nodes
* Avoid master/control-plane nodes

## Part 2: CronJobs

## 9. What Is a CronJob?

A **CronJob** runs **jobs on a time-based schedule**, similar to Linux `cron`.

Use cases:

* Backups
* Cleanup tasks
* Periodic reports

---

## 10. CronJob vs Job

| Feature          | Job | CronJob |
| ---------------- | --- | ------- |
| Runs once        | ✅   | ❌       |
| Runs on schedule | ❌   | ✅       |
| Creates Jobs     | ❌   | ✅       |

---

## 11. CronJob Schedule Format

Uses standard cron syntax:

```
* * * * *
| | | | |
| | | | +-- Day of week
| | | +---- Month
| | |------ Day of month
| |-------- Hour
|---------- Minute
```

---

## 12. CronJob YAML Example

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: backup-job
spec:
  schedule: "0 2 * * *"
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: backup
            image: backup-image
          restartPolicy: OnFailure
```

Runs:

* Every day at **2 AM**

---

## 13. CronJob Commands

```bash
kubectl get cronjob
kubectl describe cronjob backup-job
kubectl get jobs
```

---

## 14. Key Differences: DaemonSet vs CronJob

| Feature           | DaemonSet | CronJob |
| ----------------- | --------- | ------- |
| Runs continuously | ✅         | ❌       |
| One pod per node  | ✅         | ❌       |
| Time-based        | ❌         | ✅       |
| Creates Jobs      | ❌         | ✅       |

---

## 15. CKA Exam Tips ⭐

* DaemonSet → one pod per node
* Used for **system-level workloads**
* Scheduler used from v1.12+
* CronJob creates **Jobs**
* CronJob schedule uses standard cron syntax
