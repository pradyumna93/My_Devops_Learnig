
# Resource Requests, Limits, LimitRanges & ResourceQuotas

## 1. Why Resource Management Is Important

In a Kubernetes cluster:

* Each **node** has limited **CPU and memory**
* Each **pod** consumes CPU and memory to run

👉 If resource usage is not controlled:

* One pod can consume all resources
* Other pods may starve or fail
* Nodes can become unstable

---

## 2. How Scheduling Works with Resources

When a pod is created:

1. The **scheduler** checks the pod’s resource requirements
2. It looks for a node with **enough available CPU and memory**
3. If found → pod is scheduled
4. If not found → pod stays in **Pending** state

You can confirm this using:

```bash
kubectl describe pod <pod-name>
```

You may see:

```
Insufficient cpu
Insufficient memory
```

---

## 3. Resource Requests (Guaranteed Resources)

### What Is a Resource Request?

A **request** is the **minimum guaranteed** CPU or memory a container needs.

The scheduler uses **requests** to decide:

* Which node the pod can run on

---

### Pod Example with Requests

```yaml
resources:
  requests:
    cpu: "2"
    memory: "4Gi"
```

✔️ The pod will be scheduled **only on nodes** with at least:

* 2 CPUs
* 4Gi memory available

---

## 4. CPU Units Explained (Important for Exam)

| CPU Value | Meaning         |
| --------- | --------------- |
| 1         | 1 vCPU / 1 core |
| 0.5       | Half CPU        |
| 100m      | 0.1 CPU         |
| 1m        | Smallest unit   |

📌 1 CPU =

* 1 vCPU (AWS)
* 1 core (GCP/Azure)

---

## 5. Memory Units Explained

| Unit | Meaning            |
| ---- | ------------------ |
| Mi   | Mebibyte (1024 MB) |
| Gi   | Gibibyte (1024 Mi) |
| M    | Megabyte (1000 MB) |
| G    | Gigabyte (1000 MB) |

⚠️ `Gi ≠ G` (exam trap)

---

## 6. Resource Limits (Maximum Usage)

### What Is a Resource Limit?

A **limit** defines the **maximum** CPU or memory a container can use.

---

### Pod Example with Limits

```yaml
resources:
  limits:
    cpu: "1"
    memory: "512Mi"
```

---

## 7. Requests vs Limits (Key Differences)

| Resource | What Happens When Limit Is Exceeded |
| -------- | ----------------------------------- |
| CPU      | Throttled (slowed down)             |
| Memory   | Pod is killed (OOMKilled)           |

Memory **cannot be throttled**, only reclaimed by killing the pod.

---

## 8. Default Behavior (Very Important)

By default:

* ❌ No requests
* ❌ No limits

Result:

* Any pod can consume **all node resources**
* Other pods may starve

⚠️ This is **dangerous in production**

---

## 9. CPU Scheduling Scenarios (Very Important)

### Case 1: No Requests, No Limits ❌

* One pod can consume all CPU
* Other pods may starve

---

### Case 2: Limits Set, Requests Not Set

Kubernetes assumes:

```
request = limit
```

✔️ Guaranteed CPU
❌ No CPU sharing flexibility

---

### Case 3: Requests and Limits Set

* Guaranteed minimum CPU
* Cannot exceed limit

✔️ Predictable
❌ May waste unused CPU

---

### Case 4: Requests Set, No Limits ✅ (Best Practice for CPU)

* Guaranteed minimum CPU
* Can use extra CPU if available
* Fair sharing when contention happens

✅ **Recommended for CPU**

---

## 10. Memory Scheduling Scenarios

Memory behaves differently than CPU.

### Requests Set, No Limits ❌ (Risky)

* Pod can consume extra memory
* If memory pressure occurs → pod is killed

⚠️ Not recommended for memory-heavy apps

---

### Requests + Limits Set ✅

* Guaranteed memory
* Safe upper bound
* Prevents node crashes

✅ **Recommended for memory**

---

## 11. Best Practice Summary

| Resource   | Recommendation               |
| ---------- | ---------------------------- |
| CPU        | Set requests, avoid limits   |
| Memory     | Set both requests and limits |
| Production | Always set requests          |

---

## 12. Multi-Container Pods

* Requests and limits are set **per container**
* Total pod usage = sum of all containers

---

## 13. LimitRange (Namespace Defaults)

### Why LimitRange?

To ensure:

* Every pod gets default requests/limits
* Developers don’t forget to define them

📌 Applies **per namespace**

---

### LimitRange Example (CPU)

```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: cpu-limits
spec:
  limits:
  - type: Container
    default:
      cpu: "500m"
    defaultRequest:
      cpu: "500m"
    max:
      cpu: "1"
    min:
      cpu: "100m"
```

✔️ Affects **new pods only**

---

## 14. ResourceQuota (Namespace Hard Limits)

### Why ResourceQuota?

To restrict:

* Total CPU & memory used by all pods
* Prevent overuse in a namespace

---

### ResourceQuota Example

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: compute-quota
spec:
  hard:
    requests.cpu: "4"
    requests.memory: "24Gi"
    limits.cpu: "10"
    limits.memory: "32Gi"
```

✔️ Enforced at namespace level
✔️ Hard limit

---

## 15. CKA Exam Quick Notes 🧠

* Scheduler uses **requests**, not limits
* CPU is throttled, memory is killed
* Requests guarantee resources
* LimitRange → defaults
* ResourceQuota → namespace cap
* OOMKilled = memory limit exceeded
