
# Kubernetes Scheduling – Manual Scheduling

In this lecture, we explore **manual scheduling of pods** in Kubernetes and understand how scheduling works when a scheduler is unavailable or when you want full control over pod placement.

---

## How Kubernetes Scheduling Works

Every Pod has a field called **`nodeName`**.

* By default, `nodeName` is **not set**
* The Kubernetes scheduler:

  * Watches for Pods without `nodeName`
  * Selects a suitable node
  * Assigns the Pod to that node by setting `nodeName`
  * Creates a **binding object** internally

If the scheduler is **not running**, Pods will remain in the **Pending** state.

---

## Manual Scheduling Options

### 1️⃣ Manual Scheduling at Pod Creation Time (Recommended)

The **simplest** way to manually schedule a pod is by specifying the `nodeName` field in the Pod YAML file **before creating the Pod**.

### Example: Full Pod YAML with `nodeName`

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: manual-pod
  labels:
    app: nginx
spec:
  nodeName: node01
  containers:
    - name: nginx-container
      image: nginx
      resources:
        requests:
          cpu: "250m"
          memory: "128Mi"
        limits:
          cpu: "500m"
          memory: "256Mi"
```

✅ When this Pod is created, Kubernetes **skips the scheduler**
✅ The Pod is directly assigned to **node01**

Create the pod using:

```bash
kubectl apply -f manual-pod.yaml
```

---

## Important Notes About `nodeName`

* `nodeName` **can only be set at creation time**
* You **cannot modify** `nodeName` after the Pod is created
* If the node does not exist or is not reachable, the Pod will remain **Pending**

---

### 2️⃣ Scheduling an Existing Pod Using a Binding Object

If the Pod is **already created** and stuck in the Pending state, Kubernetes **does not allow editing** the `nodeName` field.

To handle this, you must manually create a **Binding object**, which mimics the scheduler’s behavior.

---

## Binding Object (YAML)

```yaml
apiVersion: v1
kind: Binding
metadata:
  name: manual-pod
target:
  apiVersion: v1
  kind: Node
  name: node01
```

⚠️ Kubernetes does **not accept YAML directly** for bindings
You must convert it to **JSON** and send it via a **POST request**.

---

## Binding Object (JSON)

```json
{
  "apiVersion": "v1",
  "kind": "Binding",
  "metadata": {
    "name": "manual-pod"
  },
  "target": {
    "apiVersion": "v1",
    "kind": "Node",
    "name": "node01"
  }
}
```

---

## Send the Binding Request

```bash
curl -X POST \
  -H "Content-Type: application/json" \
  --data @binding.json \
  http://localhost:8001/api/v1/namespaces/default/pods/manual-pod/binding
```

(Assuming `kubectl proxy` is running)

---

## Summary

| Method                 | When to Use  | Key Point                 |
| ---------------------- | ------------ | ------------------------- |
| `nodeName` in Pod YAML | New Pod      | Simplest and recommended  |
| Binding Object         | Existing Pod | Mimics scheduler behavior |
