
# Kubernetes Scheduling – Inspection & Manual Scheduling

---

## Exercise 1: Verify the Scheduler Is Running

### 🧪 Task

Check whether the Kubernetes scheduler is running in the cluster.

### ✅ Solution

```bash
kubectl get pods -n kube-system
```

Expected output:

```text
kube-scheduler-controlplane   1/1   Running
```

📌 **Exam Tip**:
If the scheduler is missing or not running, all newly created pods will remain **Pending**.

## Exercise 2: Inspect Scheduler Logs

### 🧪 Task

View the logs of the Kubernetes scheduler to check for errors.

### ✅ Solution

```bash
kubectl logs kube-scheduler-controlplane -n kube-system
```

To stream logs:

```bash
kubectl logs -f kube-scheduler-controlplane -n kube-system
```

📌 Look for:

* `FailedScheduling`
* `No nodes available`
* Resource constraint messages

## Exercise 3: Pod Stuck in Pending – Identify the Reason

### 🧪 Task

A pod named `nginx-pod` is stuck in the **Pending** state. Identify why.

### ✅ Solution

```bash
kubectl describe pod nginx-pod
```

Example event output:

```text
Warning  FailedScheduling  default-scheduler  0/2 nodes are available:
insufficient memory
```

📌 **Conclusion**:
The scheduler could not find a node with enough memory.

## Exercise 4: Manually Schedule a Pod Using `nodeName`

### 🧪 Task

Create a pod named `manual-nginx` and schedule it on `node01` without using the scheduler.

### ✅ Solution (Full YAML)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: manual-nginx
spec:
  nodeName: node01
  containers:
    - name: nginx
      image: nginx
```

Apply:

```bash
kubectl apply -f manual-nginx.yaml
```

Verify:

```bash
kubectl get pod manual-nginx -o wide
```

📌 **Key Point**:
Using `nodeName` **bypasses the scheduler completely**.

### 🧪 Task

### ❌ Expected Result


📌 **Learning**:
`nodeName` **cannot be modified after pod creation**.

## Exercise 6: Schedule an Existing Pod Using a Binding Object

### 🧪 Task

A pod named `binding-pod` is already created and Pending. Assign it to `node01`.

### ✅ Solution

#### Step 1: Create Binding JSON

```json
{
  "apiVersion": "v1",
  "kind": "Binding",
  "metadata": {
    "name": "binding-pod"
  },
  "target": {
    "apiVersion": "v1",
    "kind": "Node",
    "name": "node01"
  }
}
```

#### Step 2: Send Binding Request

```bash
kubectl proxy
```

In another terminal:

```bash
curl -X POST \
  -H "Content-Type: application/json" \
  --data @binding.json \
  http://localhost:8001/api/v1/namespaces/default/pods/binding-pod/binding
```

📌 **Exam Insight**:
Binding objects mimic what the scheduler does internally.

## Exercise 7: Identify Which Scheduler a Pod Is Using

### 🧪 Task

Find which scheduler is being used by a pod named `web-pod`.

### ✅ Solution

```bash
kubectl get pod web-pod -o yaml
```

Look for:

```yaml
spec:
  schedulerName: default-scheduler
```

If missing → **default scheduler is used**

## Exercise 8: View Scheduler Events for a Failed Pod

### 🧪 Task

Check scheduler-related events for a pod that failed to schedule.

### ✅ Solution

```bash
kubectl describe pod failed-pod
```

Example:

```text
Warning  FailedScheduling  custom-scheduler  node(s) had taints
```

📌 Shows:

* Which scheduler tried to schedule the pod
* Exact failure reason
