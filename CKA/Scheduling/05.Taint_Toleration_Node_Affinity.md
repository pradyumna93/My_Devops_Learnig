
# Kubernetes Node Selector & Node Affinity

## 1. Problem Statement

In a Kubernetes cluster:

* You have **multiple worker nodes**
* One node is **larger** (more CPU / memory)
* Some workloads (like **data processing jobs**) require **high resources**

👉 You want **only those heavy workloads** to run on the large node
👉 You want to **prevent them from running on smaller nodes**

By default:

* Kubernetes scheduler can place **any pod on any node**
* This may cause resource exhaustion on small nodes

So, we need a way to **control pod placement**

---

## 2. Ways to Control Pod Placement

Kubernetes provides **two main methods**:

1. **Node Selector** (simple)
2. **Node Affinity** (advanced & flexible)

---

## 3. Node Selector (Simple Method)

### What is Node Selector?

* A **pod-side restriction**
* Tells the scheduler:

  > “Run this pod only on nodes with specific labels”

---

## 4. Labels on Nodes

Node selectors work using **node labels**.

### Label a Node

```bash
kubectl label nodes node1 size=large
```

Now:

* `node1` has label → `size=large`

---

## 5. Pod with Node Selector

### Pod YAML Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: data-processing-pod
spec:
  containers:
  - name: app
    image: data-processor
  nodeSelector:
    size: large
```

### Result

* Pod is scheduled **only on nodes labeled `size=large`**
* In this case → `node1`

---

## 6. Limitations of Node Selector ❌

Node selectors are **very limited**:

❌ Cannot use logical conditions
❌ Cannot say:

* `large OR medium`
* `NOT small`
* `anything except small`

👉 Node Selector supports **only exact match**

---

## 7. Node Affinity (Advanced Method)

### Why Node Affinity?

Node affinity provides:

* Logical operators
* Multiple conditions
* Better control

Node affinity is the **advanced version of node selector**

---

## 8. Node Affinity Structure

Node affinity is defined under the `spec` section:

```yaml
affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - key: size
          operator: In
          values:
          - large
```

Even though this looks complex, it does the **same thing** as nodeSelector.

---

## 9. Understanding matchExpressions

Each expression has:

| Field    | Meaning        |
| -------- | -------------- |
| key      | Node label key |
| operator | How to match   |
| values   | Allowed values |

---

## 10. Common Node Affinity Operators

| Operator     | Meaning              |
| ------------ | -------------------- |
| In           | Value must match     |
| NotIn        | Value must NOT match |
| Exists       | Label must exist     |
| DoesNotExist | Label must not exist |

---

## 11. Examples of Node Affinity Rules

### Large OR Medium Nodes

```yaml
operator: In
values:
- large
- medium
```

---

### Exclude Small Nodes

```yaml
operator: NotIn
values:
- small
```

---

### Any Node with `size` Label

```yaml
operator: Exists
```

(No values needed)

---

## 12. Node Affinity Types (VERY IMPORTANT)

Node affinity has **two main types**:

### 1️⃣ RequiredDuringSchedulingIgnoredDuringExecution

* **Hard rule**
* Pod will NOT be scheduled if rule is not met

Use when:

* Pod placement is **critical**

---

### 2️⃣ PreferredDuringSchedulingIgnoredDuringExecution

* **Soft rule**
* Scheduler tries to match
* If not possible, pod runs anywhere

Use when:

* Running the pod is more important than location

---

## 13. Lifecycle States Explained

### During Scheduling

* Pod is being created
* Scheduler checks node affinity rules

### During Execution

* Pod is already running
* Node labels change later

---

## 14. What Happens If Labels Change?

Current node affinity types use:

```
IgnoredDuringExecution
```

Meaning:

* Pod **continues running**
* Even if node labels change later

⚠️ Pods are **NOT evicted** if labels are removed

---

## 15. Summary Comparison

| Feature             | Node Selector | Node Affinity |
| ------------------- | ------------- | ------------- |
| Simple              | ✅             | ❌             |
| OR / NOT logic      | ❌             | ✅             |
| Multiple conditions | ❌             | ✅             |
| Exam importance     | Medium        | High          |

---

## 16. CKA Exam Tips ⭐

* Use **nodeSelector** for simple cases
* Use **nodeAffinity** for complex rules
* Remember:

  * `Required` = strict
  * `Preferred` = best effort
* Node affinity controls **pod placement**
* Taints & tolerations control **node acceptance**
